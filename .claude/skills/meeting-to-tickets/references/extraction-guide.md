# 議事録からのアクションアイテム抽出ガイド

## 抽出の原則

議事録は「情報の海」である。その中から「実行可能なアクション」だけを正確に拾い上げるのがこのスキルの核心。

### チケットにするもの（DO）

| パターン | 例 | テンプレート |
|---------|-----|------------|
| 明示的な実装タスク | 「テロップ欄を追加する」「E2Eテストを導入する」 | new_feat |
| バグ・不具合の報告 | 「サインアップメールが届かない」「パディングがずれている」 | bug-fix |
| 改善要望 | 「UIを見やすくしたい」「2列表示にしたい」 | improvement |
| 調査・検証タスク | 「Bright Dataの費用を調べる」「マルチテナントの認証を検討」 | research |
| 決定事項で実行が必要なもの | 「Slackに移行することが決定」→ 移行作業のチケット | new_feat / improvement |
| 依頼・約束 | 「ロゴを作成して後で渡す」「必要な項目をリストアップする」 | new_feat |

### チケットにしないもの（DON'T）

| パターン | 例 | 理由 |
|---------|-----|------|
| 情報共有・説明 | 「リニアにはボードビューがある」 | アクション不要 |
| 完了済みの報告 | 「サインアップメールのバグを修正した」 | 既に完了 |
| 一般論・方針の議論 | 「アジャイルが良い」 | 具体的アクションなし |
| 雑談・余談 | 「散歩しながら参加」 | 無関係 |
| 学習・勉強会の内容自体 | 「ステータスにはBacklog, In Progressがある」 | 知識であってタスクではない |

## 粒度の判断基準

**1チケット = 1人が1つの成果物を出せる単位**

- 大きすぎる例: 「マルチテナント機能を実装する」→ 複数チケットに分割
- 適切な例: 「エンドユーザーが代理店をチーム招待できる機能を実装する」
- 小さすぎる例: 「ボタンの色を変える」→ 単独チケットにせず、UI改善チケットにまとめる

ただし、議事録の情報だけで分割判断が難しい場合は、**大きめの粒度でチケットを作り、ラベルに `needs-breakdown` を追加**する。

## プロジェクト判定のヒント

議事録内で以下のキーワードが出たら、そのプロジェクトに紐づける:

| キーワード | プロジェクト名 |
|-----------|-------------|
| YouTube Director, YT Director, 台本, 絵コンテ, サムネイル | youtube-director |
| TikTok Director, TK Director, ティックトック | tiktok-director |
| SNS Director, SNSディレクター | sns-director |
| OEM, マルチテナント, 代理店, テナント, ホワイトラベル | platform |
| Slack, Discord, リニア連携, ツール | infra |
| 法人, 登記, 投資, 契約 | business |
| 上記のいずれにも当てはまらない | general |

プロジェクト名はフォルダ名としても使われるため、英語のケバブケースで統一する。

## Priority判定の基準

テンプレート `research.md` に記載された基準を全テンプレートに適用する:

| Priority | 基準 | 例 |
|----------|------|-----|
| **Urgent** | 本番障害・ユーザーに直接影響・他の作業をブロック（即時対応） | 本番のサインアップが壊れている |
| **High** | 今週中に対応必要（サイクル内で最優先） | 期限が明示されたタスク、ブロッカー |
| **Medium** | 今サイクル内で対応（標準的なタスク） | 通常の機能追加・改善 |
| **Low** | バックログ・将来対応（余裕があれば） | nice-to-have、将来的な構想 |

## Assignee推定のルール

1. 「〜氏が〜することを約束した」→ その人がAssignee
2. 「〜氏に依頼された」→ 依頼された人がAssignee
3. 「〜氏が担当している」→ その人がAssignee
4. 専門領域から推定できる場合（例: UI/UXの話題 → デザイナー）→ 推定して「（推定）」と付記
5. 不明な場合 → 「未定」

## テンプレート記入のガイドライン

### bug-fix テンプレート
- 「症状」: 議事録から問題の説明を抽出
- 「再現手順」: 言及があれば記載、なければ「※議事録から情報なし — 要確認」
- 「期待される動作」: 文脈から推定
- 「実際の動作」: 報告された問題
- 「環境」: 言及があれば記載

### improvement テンプレート
- 「背景」: なぜ改善が必要か
- 「現状」: 今の状態
- 「改善後」: 望ましい状態
- 「やること」: 具体的な変更内容
- 「コンテキスト」: 議事録の該当箇所への参照

### new_feat テンプレート
- 「背景」: なぜこの機能が必要か
- 「やること」: 実装タスクのリスト
- 「受け入れ条件」: 議事録から読み取れる完了条件
- 「参考」: 関連するリンクや資料
- 「コンテキスト」: 議事録の該当箇所

### research テンプレート
- 「背景」: なぜ調査が必要か
- 「調査項目」: 調べるべきポイント
- 「成果物」: 期待される成果物
- 「技術的な結論・推奨アプローチ」: 議事録内で既に結論が出ていれば記載
- 「次のアクションアイテム」: 調査後に想定される次のステップ

## 重複チケットの回避

同じトピックが議事録内で複数回言及されることがある。以下のルールで重複を避ける:

1. 同じアクションが複数箇所で言及 → 1つのチケットにまとめ、すべてのタイムスタンプを `source_timestamp` に列挙
2. 関連するが別のアクション → 別チケットにし、相互参照を本文に記載
3. 判断に迷う場合 → 別チケットにして、ラベルに `possibly-duplicate` を追加
