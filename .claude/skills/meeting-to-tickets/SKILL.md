# /meeting-to-tickets — 議事録からLinearチケットのラフ案を自動生成

あなたは議事録を読み解き、プロジェクトごとのLinearチケットのラフ案を作成するエキスパートです。
長時間の会議メモから実行可能なアクションアイテムを正確に抽出し、チームのテンプレートに沿ったチケットを生成します。

## トリガー

ユーザーが `/meeting-to-tickets` を呼び出すか、以下のような依頼をした場合に起動:

```
/meeting-to-tickets workspace/linear-tasks/meeting/会議メモ.md
/meeting-to-tickets 今日の議事録からチケットを作って
議事録からタスクを抽出して
会議メモからLinearのチケットを作成して
ミーティングの内容からアクションアイテムをチケット化して
```

## ワークフロー

### Phase 1: 入力の確認と議事録の分割読み込み

1. ユーザーが議事録ファイルを指定していればそれを使用
2. 指定がなければ `workspace/linear-tasks/meeting/` 内のファイルを一覧表示し、どれを処理するか確認
3. **議事録の分割読み込み（必須）**:
   - 議事録ファイルは **必ず分割して読む**。一度に全文を読もうとしない
   - Read ツールの `offset` / `limit` パラメータを使い、**1000行ずつ**チャンクで読み込む
   - 各チャンクを読んだら、そのチャンク内のアクションアイテム候補をメモしてから次のチャンクに進む
   - 全チャンクを読み終えてから、抽出結果を統合する

4. **Explore エージェントの活用（必須）**:
   - 議事録の解析には `Agent` ツールで `subagent_type: "Explore"` を使用する
   - **1 Explore エージェント = 1000行**（チャンク単位）
   - Explore エージェントに議事録ファイルパスとチャンク範囲を渡し、以下を依頼:
     - 「このチャンク内のアクションアイテム（タスク、決定事項、依頼、課題）を抽出してください」
     - 「各アイテムについて：担当者、プロジェクト、タイプ（bug-fix/improvement/new_feat/research）、優先度を推定してください」
   - 長い議事録の場合は **複数の Explore エージェントを並列起動** して異なるチャンクを同時に解析できる
   - 各 Explore エージェントの結果を統合して、重複を排除する

### Phase 2: テンプレートの読み込み

以下の4つのテンプレートを読み込み、チケットの種類を判定する基準にする:

- `workspace/linear-tasks/templates/ticket/bug-fix.md` — バグ修正
- `workspace/linear-tasks/templates/ticket/improvement.md` — 改善
- `workspace/linear-tasks/templates/ticket/new_feat.md` — 新機能
- `workspace/linear-tasks/templates/ticket/research.md` — 調査・リサーチ

### Phase 3: Linear 既存チケットの確認（重複防止）

**チケットを作る前に、必ず Linear の現状を確認する。**

1. `linear-reader` サブエージェント（`.claude/agents/linear-reader.md`）を `Agent` ツールで起動する
2. サブエージェントに以下を依頼:
   ```
   Linear のメタデータと既存イシューを取得してください。
   - 全プロジェクトの既存イシュー（完了以外）
   - 利用可能なステータス、ラベル、サイクル、メンバー情報
   ```
3. 取得結果を保持し、以下の用途で使う:
   - **重複防止**: Phase 5 でチケットを作成する際、既存イシューと重複しないかチェック
   - **メタデータの正確性**: assignee、label、cycle は Linear 上の実際の値に合わせる
   - **プロジェクト名の整合**: Linear 上のプロジェクト名と一致させる

### Phase 4: 議事録の解析とアクションアイテム抽出

議事録を精読し、以下の観点でアクションアイテムを抽出する:

**抽出ルール（詳細は `references/extraction-guide.md` 参照）:**

1. **明示的なタスク**: 「〜をやる」「〜を実装する」「〜を修正する」「〜を調べる」など、具体的なアクションが述べられた箇所
2. **合意事項**: 「〜することが決定された」「〜という結論に至った」など、実行が必要な決定事項
3. **依頼・要望**: 「〜してほしい」「〜が必要」「〜を追加したい」など
4. **課題・問題点**: バグ報告、改善提案、調査が必要な事項

**抽出しないもの:**
- 単なる情報共有や説明（アクション不要なもの）
- 既に完了済みの報告
- 雑談・余談

### Phase 5: チケットの分類と構造化

抽出した各アクションアイテムに対して:

1. **プロジェクトの判定**: どのプロジェクトに属するかを判定（例: YouTube Director, TikTok Director, インフラ, 全体 など）
2. **テンプレートの選択**: アクションの性質に応じて4つのテンプレートから最適なものを選択:
   - バグ・不具合の報告 → `bug-fix.md`
   - 既存機能の改善・UI/UX変更 → `improvement.md`
   - 新機能の追加・新規実装 → `new_feat.md`
   - 調査・リサーチ・技術検証 → `research.md`
3. **メタデータの付与**:
   - **Priority**: Urgent / High / Medium / Low（research.md の基準に従う）
   - **Status**: 常に **Todo**（Backlog は使わない。議事録から起票するチケットはすべて即対応対象）
   - **Assignee**: 議事録から担当者を推定し、**Linear に登録されている名前**を使う（不明な場合は「未定」）。linear-reader で取得したメンバー一覧と照合すること
   - **Cycle**: 以下の優先順位で決定する:
     1. 議事録で特定サイクルが言及された場合はその名称
     2. due日付が直近アクティブサイクルの期間内に収まる場合はそのサイクル名
     3. due日付がサイクル期間を超える場合は「未定」（Linearで確認後に設定）
     4. dueが未指定の場合は直近のアクティブサイクル（linear-reader で取得）
   - **Due**: 議事録で期限が明示された場合は ISO 形式（YYYY-MM-DD）で記載。**期限の言及がない場合は直近サイクルの最終日（金曜日）を設定する**。Linear の `dueDate` フィールドに対応
   - **Label**: **Linear に存在するラベルのみ**使用する。linear-reader で取得したラベル一覧と照合すること（存在しないラベルを作らない）
   - **Type**: 使用したテンプレート名（bug-fix, improvement, new_feat, research）

### Phase 6: ラフ案の作成

`workspace/linear-tasks/tasks/` 配下に以下の構造でファイルを作成:

```
workspace/linear-tasks/tasks/
└── {project-name}/
    └── {YYYY-MM-DD}_{HH-mm}_{ticket-title-slug}.md
```

**ファイル命名規則:**
- `{project-name}`: プロジェクト名（例: `youtube-director`, `tiktok-director`, `infra`, `general`）
- `{YYYY-MM-DD}`: 作成日（議事録の日付ではなく、チケット作成日）
- `{HH-mm}`: 作成時刻
- `{ticket-title-slug}`: チケットタイトルの短縮英語スラッグ（例: `fix-signup-email-bug`）

**各チケットファイルのフォーマット:**

```markdown
---
title: チケットタイトル
project: プロジェクト名
type: bug-fix | improvement | new_feat | research
priority: Urgent | High | Medium | Low
status: Todo
assignee: 担当者名（Linearに登録されている名前）
cycle: サイクル名または未定
due: YYYY-MM-DD または未定
labels:
  - ラベル1
  - ラベル2
source: 議事録ファイル名
source_timestamp: 該当箇所のタイムスタンプ（あれば）
created: YYYY-MM-DD HH:mm
---

（ここからテンプレートに沿った本文）
```

### Phase 7: レビュー提示

全チケットの作成が完了したら、以下の形式でサマリーを提示する:

```
## 作成したチケット一覧

### {プロジェクト名1}
| # | タイトル | タイプ | Priority | Assignee | ファイル |
|---|---------|--------|----------|----------|---------|
| 1 | ...     | ...    | ...      | ...      | ...     |

### {プロジェクト名2}
| # | タイトル | タイプ | Priority | Assignee | ファイル |
|---|---------|--------|----------|----------|---------|
| 1 | ...     | ...    | ...      | ...      | ...     |
```

その後、ユーザーに確認を求める:

```
上記のチケットラフ案を作成しました。

次のアクションを選んでください:
1. **このまま確定** — 修正不要であればこのまま残します
2. **個別に修正** — 特定のチケットを修正します（番号を指定）
3. **チケットを追加** — 見落としているアクションアイテムがあれば追加します
4. **チケットを削除** — 不要なチケットを削除します（番号を指定）
5. **Linearに投入** — Linear MCPを使ってチケットを直接作成します
```

### Phase 8: Linear投入（オプション）

ユーザーが「Linearに投入」を選択した場合:

1. Linear MCPの `list_teams` でチーム情報を取得
2. `list_issue_statuses` でステータスの選択肢を確認
3. `list_issue_labels` で利用可能なラベルを確認
4. `list_cycles` でアクティブなサイクルを確認
   - `type: "current"` で空が返る場合は `type: "next"` で再取得する（サイクル開始日が未来の場合に起きる）
5. 各チケットを `save_issue` で作成（**下記の注意事項を必ず守ること**）
6. 作成結果（Issue IDとURL）をサマリーに追記

## 重要な注意事項

- 議事録は非常に長い場合がある（数千行）。**必ず分割で読むこと**（1000行ずつ）。一度に全文を読もうとしない
- 議事録の解析には **Explore エージェントを必ず使用**する。1エージェント = 1000行。長い議事録は複数の Explore エージェントを並列起動して効率化する
- **チケット作成前に必ず `linear-reader` サブエージェントで Linear の既存チケットを確認**し、重複を避ける
- 1つの議題から複数のチケットが生まれることがある — 粒度は「1チケット = 1つの実行可能なアクション」
- 担当者の推定は議事録の文脈から行うが、確信が持てない場合は「未定」とする
- テンプレートの各セクションは、議事録から得られる情報で埋められるだけ埋める。情報不足のセクションは「※議事録から情報なし — 要確認」と明記する
- チケットタイトルは日本語で、簡潔かつ具体的に（例: 「サインアップメールのUI/UXを改善する」）

## Linear投入時の既知の落とし穴

### ラベルのグループ制約（重要）
Linearのラベルには **Area** グループと **Type** グループがある。**同じグループのラベルを1つのIssueに複数付けることはできない**。

- **Areaグループ**（1つだけ選ぶ）: メール通知, 認証, OEM基盤, アナリティクス, 決済, UI/UX, 動画生成, サムネイル, 競合分析, エディタ, AI
- **Typeグループ**（1つだけ選ぶ）: テスト, インフラ, リサーチ, バグ, 改善, 機能追加, 機能開発, 機能改善, 機能バグ, 要件定義

チケットファイルに複数のAreaラベルが書かれていても、投入時は **最も関連性の高い1つに絞る**。違反するとエラーになり、並列実行中の他のIssue作成もすべてキャンセルされる。

### save_issue の並列実行リスク
複数のIssueを並列で `save_issue` する場合、1件でもエラーになると他の並列呼び出しもキャンセルされる。ラベルの検証など事前に失敗しうる項目は**投入前に確認**すること。もしエラーが発生した場合は、成功したIssue IDを記録してから未作成分だけ再実行する。

### list_cycles の current/next 問題
サイクルの開始日が「今日以降」の場合、`type: "current"` では空リストが返る。この場合は `type: "next"` で再取得すること。
